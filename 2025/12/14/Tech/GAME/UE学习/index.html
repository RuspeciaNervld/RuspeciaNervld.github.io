CPP

官方的命名方式

A开头表示游戏对象
U开头表示引擎对象（组件资源等），AActor的祖宗UObject也是U
E开头表示枚举
F开头表示基础功能类，例如string vector

委托

UE中很多函数都是通过委托来使用，例如碰撞产生的函数

  • 委托分为动态（可以与蓝图交互）和静态，添加委托时使用AddDynamic / AddUObject
  • 定义的时候用DECLARE_(DYNAMIC_)MULTICAST_DELEGATE_XXParam
  • 一般动态的只能take一些简单的参数，如果复杂建议定义UStruct然后再传入
  • 一些跟引擎深度交互的（可以在蓝图用）的方法，在c++中最好注释UFUNCTION()

单例模式

UGameInstance 是 Unreal 官方推荐的“单例模式”实现，​既保留了单例的核心优势（全局唯一访问），又规避了手动管理风险。应当继承这个类实现单例。

常用override

UI里面的开始和结束函数是NativeDestruct(),需要override
UI里面如果使用组件需要引入头文件例如：”Components/TextBlock.h”

与Unity不同的是

这里的世界视图窗口并不灵活，父子级关系比较固定，且不容易给子组件加入代码。这就需要我们全部由根c++文件进行管理。

UE里蓝图是最接近预制体的存在，但是他只能反复继承，无法进行组合使用。

比如我要实现在特定相对位置生成武器（可以是不同类型的），在Unity里我会用预制体预先设计好不同的武器，但是UE里蓝图就比较相当于预制体，我会创建很多不同武器的蓝图

  • 在Unity管理预制体我会用GameObject列表
  • 在UE中管理蓝图则应该用STSubclassOf<AActor>，因为AActor是能放置在场景中的
  • 在Unity里面我们用Instantiate来实例化
  • 在UE里面我们用Word->SpawnActor
  • 在Unity代码中存储预制体是有意义的，因为你完全知道他的脚本结构
  • 在UE代码中存储蓝图没有意义，因为你无法在C++处调用其方法，除非设计蓝图实现接口或者将其强制转为父类调用（STSubclassOf的时候以父类C++也可)，或者广播消息让他执行。

问题汇总

打包问题

一定要安装core net 3.1
一开始拿到vs项目需要去生成我们对应项目名的
关闭livecoding然后手动编译再继续
平时开发的时候就可以打开livecoding然后ctrl+alt+f11来快速编译
不要用VS2022的build
如果还是没有加载，那就关闭编辑器，删除对应的intermediate, binaries(比如在plugins下)，然后ctrl+b，然后再打开编辑器，再编译

建议打开项目文件夹，一旦出现live coding提示编译没用，先试试非live coding，还不行就关闭VS和编辑器重新打开，并使用非live coding

另外注意蓝图会覆盖我们代码设置的变量值（例如构造函数里设置的），所以要改蓝图的值

打包地图问题：
项目设置->搜索 “map”->打包版本中要包括的关卡列表。

创建c++类之后文件夹没有显示出来：
可以先把c++里面的报错解决了然后再编译一次（无需关闭编辑器）

Editor问题

  • 如果崩溃，可以看到是不是代码问题，如果是，一般是空指针导致。
  • 否则编译导致，删光可删除的文件夹（包括plugins里面），可额外删一下Config目录
  • 如果在引擎中没有定义了的c++类，那就运行一次编译指令就好（IDE或live coding）
    • 此时有丢失父类的蓝图，右键资源进行reload
    • 去编辑器偏好设置里左侧加载和保存中勾选上启动时强制编译force compilation。（一劳永逸。）

IDE问题

  • 如果解析出问题，要右键生成vs文件，然后再自己打开IDE
  • C++类是IDE影响UE，而不是UE影响IDE，所以重新编译即可显示到UE中
    • 在IDE里重名C++类和文件名，无法立刻作用到UE，继续使用有bug，及时重启
  • 插件要加入到Build文件编译列表
  • C++修改了构造函数，但是UE需要reload才能有效
  • Attach to component 和 Attach to actor 和 SetupAttachment 和 AttachActor的区别
    • Attach to component 和 Attach to actor附加到的目标粒度不同，component可以附加到比如mesh这种组件
    • Setup用于构造函数阶段
    • AttachActor是反向的，即调用者给自己加一个Actor子类

BUG

  • cpp构造函数里设置的属性，蓝图有可能会覆盖

  • 回调函数和委托需要设置为UFUNCTION

  • 有时候可能没有传入资源导致传入值没变化hud只调用一次

  • 传递给动画机的一些变量名都是以b开头的，比如bAiming

  • 如果遇到了奇怪的问题一般都是c++类初始化有错导致的，看看是哪个组件丢失了，最好写的时候习惯性加上log

  • 网络封装带来的bug

    比如说GetBaseAimRotation().Yaw不起作用，那么我们就自己手动同步这个值，让这个值在locally的地方更新并同步

  • 子弹速度过快会导致近距离特效看不到，或者一些同步丢失

  • 用OnRep_ReplicatedMovement() override来做

  • 有可能因为有些函数在初始化的时候被调用了导致某些值被设置回去了，可以打印看看

  • 如果出现场景问题，可以删除gamemode重新配置

  • 生成方式
    -

  • 发现replicated变量的绑定方法没有调用，看看是不是设置了replicatedUsing = xxx

  • 出现了一些objectwarning，可以关闭ue，重新从rider运行

  • 重生之后会调用这个，所以用这个来做一些操作

  • 在.h文件中用class来定义就可以防止循环依赖

  • 给指针引用加上UPROPERTY()就不会出现未定义crash

动画系统

动画序列

就是各个骨骼在每一帧的相对运动信息

root motion 是将根骨骼位移应用到骨骼
force root lock 是将根骨骼限定在胶囊体内
force root lock 仅仅是在 root motion没启用的时候，用于让动画不要移动出胶囊体范围的，也就是让动画一直在原地播放，即使动画拥有根骨骼的运动数据

ABP 动画蓝图

  • 可以先做一个通用的C++动画类，用于暴露数据给动画蓝图
  • 然后创建一个不带骨架的动画蓝图模板来继承这个C++类
  • 然后从这个模板中创建动画蓝图实例，才是真正的角色

BS blend space

一般输入是速度，用一维的就够了

蒙太奇

  • 右键动画序列创建蒙太奇
  • 一般作为character的成员，由character通过PlayAnimMontage调用（当然，用AnimInstance的Montage_Play也可以）
  • 插槽是一种用于分类动画的栏目，只有在动画蓝图里使用插槽，才能播放对应动画
  • 插槽里面可以放置通知，通知的处理通过在AnimInstance里定义一个AnimNotify_xxx开头的UFUNCTION()实现

判断角色

  • Pawn拥有GetLocalRole和GetRemoteRole的方法
  • LocalRole中，客户端看自己是AutonomousProcxy，看别人是Simulated，服务器看大家都是Authority
  • RemoteRole中，客户端看所有都是Authority，服务器看自己是Autonomous，看别人是simulated
  • HasAuthority()​：用于服务器逻辑，确保关键操作由权威端执行，等价于LocalRole = Authority
  • IsLocallyControlled()：用于客户端逻辑，区分本地玩家与远程玩家，等价于RemoteRole = AutonomousProcxy || LocalRole = AutonomousProcxy，但是实际上通过GetOwner来判断控制器所有权

基本思路

  • 在本地进行操作之后，再通过RPC操作使得这个操作被服务器认可
  • 一旦服务器上做了这个操作，而这个修改的值又是Replicated，就可以无痛同步

游戏元素的关系

UE材质

ReflectionVector 是从摄像机发射一个向量，这个向量经过面反射后到达的方向

顶点法向和像素法向的区别：

  • 顶点法向就存储在顶点上，对于面上的法向值，是差值得到的
  • 像素法向存储在像素中，每个点都可以不一样，需要配合法线贴图使用，赋予了法线贴图之后才真正和顶点法向产生区别
  • 顶点法向不管有没有法线贴图都不变，因为是从模型计算出来的。
