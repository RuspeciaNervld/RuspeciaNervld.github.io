开始整理一下Service-Forge

首先这是一个agent时代的自动化工作流系统

也就是包含agent和工作流两方面的内容

让agent来写工作流，这个是目前的方案，后续可能会进一步agentic RL，让AI在写工作流这一块更擅长。

目前来说主要实现的是框架层面的东西。

首先核心是使用K8s集群来做容器编排，然后我还写了一个服务中心用于方便的部署和管理所有的服务，

那你有没有真的做过一个agent:?

这个东西的意义在于让开发者在写服务的时候不用去关心底层的日志系统、负载均衡、鉴权路由】数据库存储、知识库构建等等，比较类似于MCP服务的东西，但是

我们认为工作流是比较适合于变化量不大但是需求比较高的的场景，比如说记日程、比如说写

xxx也是优化了几个版本吧，等等之类的话术。。。。

需要把项目总结成文章。

  • 这个项目算是给实验室项目做一个技术底座，然后也非常方便后续的扩展和升级，以及更多功能的接入。
  • 我们是有一个智能戒指，可以气息唤醒说话，第二大脑算是给这个硬件订做的一个软件平台把，但是他也能独立使用。
  • 我们用的是K8s来做整个服务集群，这样比较方便后续扩展多台服务器，方便做环境隔离服务管理等等。
  • 所以我还写了一个service-center，并且提供了一个命令行工具来和他交互，这样就能很快把服务构建好后的压缩包发送到服务器来进行一个部署，也可以标记版本和环境等信息。
  • 配置中心就是借助K8s里的那个ConfigMap来实现的
  • 然后就是所有服务的基础框架，那个service-forge，是一个基于DAG的工作流服务运行工具，提供了多种触发器，比如https的、kafka的等等。
  • 然后作为开发者其实要写的代码只有核心逻辑成分，像调用大模型、日志记录、数据库维护、用户管理、token记录等等都放在这个库里，在节点运行的过程中统一自动进行。
  • 比如我们在调用大模型这一块统一走的是LiteLLM，他封装了目前的比较全面的大模型调用api以及费用等信息。
  • 用DAG主要的好处就是可以比较清晰的知道节点间的依赖关系，节点之间通过边连接，有input和output节点
  • workflow是通过yaml格式存储的，首先是解析成对象格式，然后构建节点，把他们之间通过边连接起来，构建好workflow之后通过一次扫描来看是不是有环存在，存在的话就验证失败。
  • 参考langgraph实现了基于状态机的中央状态管理，最重要的是多agent架构，结合workflow具有可控性的特点，就是一个任务比如说一个我定一个日程是帮我整理文件，我可以让大模型直接帮我做，但是我并不确定这个他做的过程是怎么样的，是不是符合要求，以及他下一次做是不是还是同样的路径，是否可以量化而不是每次都靠玄学。有一个workflow做兜底至少能限制住大模型跑偏。
  • 分层架构，这样可以让ai先专注于写工作流，因为工作流是比较关键的，然后再去写node，node其实只是实现，并且我们框架还可以在连接节点，处理节点间输入输出这块再做优化。比如说把输入输出都是流式的节点串成一个pipeline这样。
  • 我们设计的是工作流主要是agent之间的协作，上下文管理（这个好像变成了state）
  • 节点就是提示词的细化，通过单元测试提高命中率。
特性维度 LangGraph 的设计理念 Claude Code 的真实架构印证
核心架构 图式结构：通过节点（处理单元）和条件边（动态路由）构建灵活的、非线性的工作流，支持循环和分支 。 多Agent并行协作：采用主Agent领导多个Subagent的架构。在执行任务时，它会并行启动多个Subagent（如文件搜索、规划专家），分别探索代码库的不同部分，最后汇总结果 。这正是LangGraph所擅长的复杂图式路由。
状态管理 中央状态机：一个全局的State对象在节点间传递和更新，确保上下文连贯，是实现“记忆”和“时间旅行”的基础 。 智能上下文压缩与缓存：Claude Code 通过 Claude.md 文件管理项目级的长效记忆 。更厉害的是，其底层实现了高达 92% 的提示词前缀复用率，本质上是在协议层对“中央状态”进行了极致的工程优化，以实现长对话的连贯和低成本。
工具系统 工具即节点：将外部工具（如搜索引擎、计算器）作为图中的一个节点，由模型动态决策是否调用。 MCP 协议：通过 MCP（模型上下文协议）服务器，将文件系统、Shell命令、GitHub、甚至数据库等外部世界“工具化”，供Agent按需调用 。这种即插即用的工具生态，正是LangGraph“工具即节点”思想的规模化实现。
流程控制 动态决策：通过条件边让模型根据当前状态决定下一步走向（是调用工具还是直接回答）。 ReAct 循环与规划模式：从追踪日志可以看到，其 Subagent 内部普遍采用 ReAct（思考-行动-观察）循环 。同时提供了Plan模式，让Agent在执行前先输出计划，经用户确认后再动手，实现了高级的人机协作控制流 。
工程特性 生产就绪：内置节点缓存、检查点、断点续跑、人工介入等，为复杂应用的稳定性和可调试性而生。 企业级能力：拥有精细的权限管理（如Bash命令的白名单/黑名单）、沙箱模式、会话持久化与恢复（--resume参数）、以及详尽的流式输出和终端UI渲染。这些都是LangGraph生产级理念在工业界的最高标准体现。
如果阶段之间分的好的话，前面一个阶段收敛了，后面的阶段就可以只需要很少的之前的上下文信息，即可完成任务。这也是为什么工作流可以有比较好的效果。
