Spring容器启动过程/bean生命周期

  1. 加载Bean定义 → 2. 实例化（创建原始对象） → 3. 属性填充（注入依赖） → 4. 初始化前处理 → 5. 初始化（执行自定义逻辑） → 6. 初始化后处理（生成代理） → 7. Bean就绪使用 → 8. 销毁（容器关闭）
阶段 1：Bean 定义加载（容器启动第一步）
  • 核心操作：Spring 扫描 @Component、@Bean 等注解 / XML 配置，将 Bean 的元信息（类名、作用域、依赖、初始化方法等）加载到 BeanDefinition 中，存入容器的「Bean 定义注册表」。
  • 通俗理解：相当于工厂拿到 “产品图纸”，还没开始生产产品。
阶段 2：实例化（Instantiation）—— 创建 “空对象”
  • 触发时机：容器启动时（非懒加载单例）或首次使用 Bean 时（懒加载）。
  • 核心操作：通过反射执行 Bean 的构造方法，创建原始对象（仅完成 new XXX()，无属性、无业务逻辑）。
  • 关键说明：
    • 这一步只创建对象 “外壳”，比如 UserService 实例化后，@Autowired 注入的 DataSource 还是 null；
    • 如果是 CGLIB 代理的 Bean，实例化阶段会先创建子类代理的 “空对象”（但核心代理逻辑在后续阶段）。
阶段 3：属性填充（Populate）—— 给对象 “装配件”
  • 触发时机：实例化后立刻执行。
  • 核心操作：Spring 解析 @Autowired、@Resource、XML 的 <property> 等，将依赖的 Bean 注入到当前 Bean 的属性中。
  • 关键说明：
    • 这是实例化和初始化之间的 “关键桥梁”—— 必须先填属性，否则初始化时用到依赖会报空指针；
    • 循环依赖的解决也发生在这个阶段（通过 “早期单例池” 提前暴露对象）。
阶段 4：初始化前处理（PostProcessBeforeInitialization）
  • 触发时机：属性填充完成后，初始化前。
  • 核心操作：执行所有 BeanPostProcessor 的 postProcessBeforeInitialization 方法。
  • 典型场景：
    • 解析 @PostConstruct 注解，执行标注的方法；
    • Spring 内置的处理器（如 AutowiredAnnotationBeanPostProcessor）完成最后一步依赖校验。
阶段 5：初始化（Initialization）—— 给对象 “赋予生命力”
  • 触发时机：初始化前处理完成后。
  • 核心操作（按执行顺序）：
    1. 执行 InitializingBean 接口的 afterPropertiesSet() 方法；
    2. 执行自定义的初始化方法（如 @Bean(initMethod = "init") 或 XML 的 init-method）。
  • 关键说明：这一步是开发者自定义初始化逻辑的核心入口（比如初始化连接池、加载配置文件）。
阶段 6：初始化后处理（PostProcessAfterInitialization）—— 生成最终对象
  • 触发时机：初始化完成后。
  • 核心操作：执行所有 BeanPostProcessor 的 postProcessAfterInitialization 方法。
  • 典型场景：
    • AOP 动态代理的核心生成时机（无循环依赖时）：AbstractAutoProxyCreator 在此阶段检查 Bean 是否需要代理，生成 JDK/CGLIB 代理对象；
    • 最终存入单例池（singletonObjects）的是这个阶段处理后的对象（代理对象或原始对象）。
阶段 7：Bean 就绪（Ready）—— 可被使用
  • 核心状态：Bean 完成所有流程，存入单例池，对外提供服务（如被其他 Bean 注入、通过 context.getBean() 获取）。
  • 作用域差异：
    • 单例 Bean：全程只执行一次生命周期，容器启动后一直存在；
    • 原型 Bean：每次 getBean() 都会重新走一遍完整生命周期。
阶段 8：销毁（Destruction）—— 容器关闭时清理
  • 触发时机：容器关闭（如 context.close()）时（仅单例 Bean 执行，原型 Bean 由开发者自行管理）。
  • 核心操作（按执行顺序）：
    1. 执行 @PreDestroy 注解标注的方法；
    2. 执行 DisposableBean 接口的 destroy() 方法；
    3. 执行自定义的销毁方法（如 @Bean(destroyMethod = "destroy") 或 XML 的 destroy-method）。

三级缓存解决带有动态代理的循环依赖问题

  • Spring 对 Bean 生命周期的定义。对象都应该在创建之后再进行动态代理，而不是单纯的实例化以后就急着进行代理，如果循环依赖就是没办法的事

@AutoWired

用在Spring管理的Bean对象内，如加了@Component / @Service / @Controller / @Repository 等注解的。
用法：

  • 推荐用在构造函数上
  • Setter方法上
  • 成员变量上（不推荐）
  • 用在普通方法上（自动注入参数+自动执行）
  • 用在方法参数上（其实没啥用，spring调用这个方法的时候会自动注入，一般是构造函数，但是现的spring根本不需要手动autowired就会自动注入+调用构造函数）

只有一个构造器的类，构造方法上可以完全不写 @Autowired，Spring 会自动默认注入。

与@Resource（java标准库提供）

@Autowired 按类型找
@Qualifier(“userDaoImpl”) —— 多个类型用这个指定名字，比如实现类AClass的默认bean名是aClass，bean注解的括号里也可以手动给名字，例如 @Service(“service-name”)
@Resource 默认按名字找，不能用在构造方法、普通方法、字段上。
@Resource(name = “userDaoImpl”)

  • 构造注入 → 用 @Autowired（或省略）
  • 字段注入 → 推荐 @Resource
