生产者业务代码，实际底层就是发送了对消费者不可见的消息

一些概念

  1. Queue / Topic

    • 消息队列的 “主题”，业务上的一个频道。

  2. Partition（分区）/ Queue（队列）

    • Topic 下面的最小存储单元
    • Kafka 叫 Partition
    • RocketMQ 叫 Queue
    • 一个 Topic 可以分成多个分区。

  3. 消费者组（Consumer Group）

    • 一组相同逻辑的消费者，共同消费一个 Topic
    • 同一个消费者组内：一条消息只会被一个消费者消费

  4. 消费位点（Offset）

    • 标记消费到哪条消息，存在 Broker 或 ZK/Kafka 内部。

面试题

1. 为什么要用消息队列？

三大核心：

  1. 异步：不用同步等待，提升响应速度
  2. 解耦：服务之间不直接强依赖
  3. 削峰：扛住流量洪峰，保护下游系统

扩展：最终一致性、广播、错峰流控。

2. 消息队列有什么缺点？

  1. 系统复杂度变高
  2. 一致性问题（需要最终一致性方案）
  3. 可用性依赖：MQ挂了可能影响业务
  4. 可能出现重复消费、消息丢失、消息乱序

3. 如何保证消息不丢失？（三面高频）

生产端 + 存储端 + 消费端 三段保障

  1. 生产者

    • 同步/异步确认（ack）
    • 失败重试
    • 落库 + 定时任务重试（可靠生产）

  2. Broker 存储

    • 持久化
    • 多副本（同步刷盘、同步复制）
    • 主从同步保证

  3. 消费者

    • 先执行业务，再确认 ack
    • 禁止自动 ack

4. 如何解决消息重复消费？（幂等性）

核心：消费必须幂等，重复消费结果一样。

常见方案：

  1. 唯一消息 ID + 去重表（Redis/DB）
  2. 数据库唯一约束
  3. 状态机（已处理就跳过）
  4. 业务天然幂等（insert 唯一键、update 同值）

5. 如何保证消息顺序？

场景：订单创建 → 支付 → 发货 必须顺序。

方案：

  1. 全局顺序：单线程、单分区、单消费者（性能差）
  2. 局部顺序（推荐）
    • 同一业务 ID（如订单ID）哈希路由到同一个队列/分区
    • 同一个分区内天然有序
    • 消费者单线程消费该分区

6. 消息积压怎么解决？

  1. 先看为什么积压：
    • 消费者挂了/慢
    • 生产流量突增
  2. 快速处理：
    • 增加消费者（不超过队列数/分区数）
    • 临时扩容队列/分区
    • 消费逻辑优化（异步、批量、减少IO）
  3. 极端：
    • 先转发到死信队列/临时队列，事后修复重放

7. 死信队列是什么？

消息变成死信的情况：

  • 消息被拒绝
  • 过期
  • 队列满

死信队列：

  • 专门存“无法正常消费”的消息
  • 人工排查、修复后重放

8. 延时队列/延迟队列使用场景

场景：

  • 订单超时未支付，自动取消
  • 收货后7天自动确认
  • 短信/通知延迟发送

实现：

  • RabbitMQ：TTL + 死信队列
  • RocketMQ：自带延迟等级
  • Kafka：用时间轮/外部定时

9. Kafka、RocketMQ、RabbitMQ 怎么选？

  • Kafka：高吞吐、日志、流处理、大数据
  • RocketMQ：金融/电商，可靠、事务消息、延迟消息
  • RabbitMQ：轻量、易用、灵活路由、中小项目

二、场景题（面试最爱问）

场景1：订单创建后，扣减库存、通知物流、发短信，如何用MQ设计？

  1. 订单服务写完订单 → 发 MQ
  2. 库存服务、物流服务、通知服务分别订阅消费
  3. 优点：解耦、异步、不阻塞主流程
  4. 保证：可靠发送 + 消费者幂等

场景2：如果MQ突然挂了，怎么办？

  1. 业务降级：核心流程同步，非核心降级
  2. 生产者：消息先落库，定时任务重试发送
  3. 高可用：集群、主从、多机房

场景3：如何实现分布式事务？

主流方案：

  1. 可靠消息最终一致性（最常用）
    • 本地事务 + 消息表
    • 定时任务轮询未发送消息
  2. 事务消息（RocketMQ）
    • 半消息 → 执行本地事务 → 确认提交/回滚
  3. TCC（性能差、侵入强）

场景4：百万流量突洪，如何用MQ抗住？

  1. MQ 削峰，前端流量全部扔进MQ
  2. 消费者按自己能力慢慢消费
  3. 避免数据库被打垮
  4. 配合限流、降级

场景5：消费失败无限重试，怎么处理？

  1. 重试一定次数（如3次）
  2. 仍失败 → 扔死信队列
  3. 人工排查 + 修复后重放
  4. 避免死循环打垮系统

三、简短背诵版（面试前速记）

  • 作用：异步、解耦、削峰
  • 三大问题：丢失、重复、乱序
  • 不丢失：生产ack、broker持久化、消费后ack
  • 重复消费：幂等（唯一ID、去重表）
  • 顺序：同ID路由同分区，单线程消费
  • 积压：加消费者、优化消费、临时扩容
  • 死信：失败/过期/拒收 → 死信队列 → 人工重放
  • 分布式事务：可靠消息最终一致性 / 事务消息
