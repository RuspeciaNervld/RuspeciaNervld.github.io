消息可靠性

1. 生产者不丢消息
  • Kafka：acks=all + 生产者幂等 + 重试（无限重试 + 指数退避）。
  • RabbitMQ：publisher confirm + mandatory + 客户端重试。
  • RocketMQ：同步发送（sync）+ 失败重试 + 事务消息。
2. Broker 不丢消息
  • 持久化：队列 / 主题持久化 + 消息持久化（delivery_mode=2）。
  • 刷盘策略：同步刷盘（可靠性高）、异步刷盘（性能高）。
  • 副本机制：多副本（Kafka ISR、RocketMQ 主从、RabbitMQ 镜像队列）。
3. 消费者不丢消息
  • 禁用自动 ACK：手动提交（Kafka 手动 commit、RabbitMQ basicAck、RocketMQ 手动 ack）。
  • 先处理后提交：处理成功再发 ACK，否则 Broker 重投。

重复消费与幂等

1. 为什么重复？
  • 生产者重试、消费者超时重投、ACK 丢失、网络抖动。
2. 幂等方案（三种）
  • 唯一 ID + 去重表：消息加唯一 ID，消费前查 Redis/DB，已处理则跳过。
  • 业务幂等设计：状态机、乐观锁、唯一索引、增量更新。
  • 分布式锁：消费前加 Redis 锁，处理完释放，避免并发重复。

消息积压怎么办？

排查消费能力，扩容消费者，优化消费逻辑，临时扩容队列，死信队列兜底。

企业级的RAG全链路

  • 文件上传
    • 需要前端上传对整个文件的MD5，然后把文件切成chunk，一个个上传。收到第一个chunk的时候要记录一个文件主记录，标记为上传中。
    • 收到的分片文件直接存入MinIO，最后合并用的是 MinIO compose
    • 如何记录进度：通过redis的bitmap数据结构存储每个chunk的上传情况。
    • 端点续传：客户端查已经上传的分片列表，然后继续上传剩下的。
  • 上传成功后异步处理

实时通信架构

webRTC?
是一个支持浏览器和移动应用进行实时音视频通信的开源项目，无需安装插件，通过简单的JavaScript API 即可实现P2P通信。
WebRTC 不是单一协议，而是一个协议栈/框架，包含多个协议协同工作：

层级 协议 作用
应用层 JavaScript API 开发者接口
信令层 自定义/WebSocket/SIP/XMPP 协商SDP/ICE（WebRTC不规定信令协议）
媒体传输 SRTP (Secure RTP) 加密音视频传输
数据通道 SCTP over DTLS 可靠/不可靠数据传输
安全层 DTLS (Datagram TLS) 密钥交换，为SRTP/SCTP加密
传输层 UDP (主要) + TCP fallback 媒体走UDP，信令可走TCP
网络层 ICE/STUN/TURN NAT穿透框架
WebRTC 是面向连接的，但基于UDP实现（这一点常被误解）。
特性 WebRTC 对比
连接建立 需要握手（ICE + DTLS） 类似TCP的”连接”概念
底层协议 UDP (用户数据报协议) 非TCP
可靠性 可选：SRTP(不可靠) / SCTP(可靠) 应用层控制
有序性 可选配置 数据通道可设 ordered: true/false

异步任务管道

  • 生产这配置：
    • acks=all，确保消息所在分区的所有同步副本都写入成功才认为成功
    • 首次消费失败后开始重试、每次重试间隔 3 秒、加上首次消费，一共最多尝试 5 次、仍失败就投递到 DLT
    • factory.setTransactionIdPrefix(“kafka-tx-“); Kafka 事务 = 保证 “发消息” 和 “数据库操作” 同时成功 / 同时失败。
